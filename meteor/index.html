Meteor Framework

What, Why, Wat

High-level Architecture

It's Real-time

  • Collection Synchronization
  • Reactive data sources
  • Reactive templates

The Good

  • Looks cool
  • Might be good for certain types of apps
  • Fibers on the server (no callbacks, yay!)

The Bad

  • Always connected
  • Collection sync
  • Reactivity
  • Minimongo
  • Client
  • Server
  • Performance
  • Ecosystem

Always Connected

  1. Meteor applications are not crawlable
  2. No server-side templates
  3. Headless webkit on the server to render pages (duh)

Collection synchronization

  1. Paging not possible without ugly hacks
  2. Limited API
  3. No rate limiting
  4. Wasted bandwidth (mobile anyone?)

Bad API example

// Keep track of how many administrators are online.
var count = 0;
var query = Users.find({admin: true, onlineNow: true});
var handle = query.observeChanges({
  added: function (id, user) {
    count++;
    console.log(user.name + " brings the total to " + count + " admins.");
  },
  removed: function () {
    count--;
    console.log("Lost one. We're now down to " + count + " admins.");
  }
});

Reactivity

  1. Feels bolted-on in both client and server
  2. Mongo is not reactive - polling all the way
  3. Using non-reactive templating engine feels awkward

Minimongo

  1. No indexes
  2. No aggregations
  3. No map-reduce
  4. Slooooow

Client

  1. Flat variables namespace (Meteor.session)
  2. Flat templates namespace (Meteor.templates)
  3. No built-in routing
  4. No two-way data-binding
  5. Unstable DOM
  6. Slow

Meteor.session


		 Meteor.session.set('mymodule.submod.value', 10);

Client: Continued

  1. Meteor.autorun() handle leaking
  2. Meteor-router double view triggering
  3. Lots of special cases in templates

Client: Slow

  1. Reactive updates re-render whole template
  2. Need to put hints everywhere to isolate contexts
  3. Minimongo is not fast either

Server

  1. Flat methods namespace (Meteor.methods)
  2. Ain't fast (minimongo + reactivity = bottleneck)
  3. No command-like API (no scripts!)
  4. No server-side templates
  5. Slow restart cycle

Performance

  1. 600 req/s on AWS large for REST API
  2. 3 second Chrome DOM lockup with 12k items in collection on Core i7
  3. Any data manipulation on client is big no-no

Ecosystem

  1. Own package manager
  2. Didn't play nice with npm before, had to wrap into own packages
  3. Young platform

Real experience

Talkalytics

  1. Customers
  2. Orders
  3. Real-time analytics

Problems

  1. Horrible client-side and server-side performance
  2. Amount of data during initial sync is overwhelming
  3. Browser DOM lockups
  4. Mobile clients were dying

Changes

  1. Got rid of collection synchronization
  2. REST-like API to work with data
  3. Client was dumbed down, minimongo not used
  4. Used MongoDB directly on server
  5. Reactivity is reduced to bare minimum

TLDR

  1. Minimongo out
  2. Reactivity out
  3. Left with Backbone-like app on top of Meteor
  4. Still share always-connected app shortcomings

Quote from meteor.com


Clean, powerful data synchronization.

Write your client code as if it were running on the server and had direct access to the database. No more loading your data from REST endpoints.

Conclusions

  1. Meteor is not mature
  2. Meteor is slow
  3. Meteor is not for mobile
  4. Meteor is not search engine friendly
  5. Core features don't work well if you have data
  6. No control
